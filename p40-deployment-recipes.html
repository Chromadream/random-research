Deployment Recipes

Production-ready configurations for the 4× NVIDIA P40 (96 GB VRAM) inference server. All recipes use pascal-pkgs-ci for vLLM or llama.cpp with CUDA. See the main build guide for hardware selection, cooling, and software stack details.

  • Recipe A: Llama 3.1 70B
  • Recipe B: DeepSeek-V4-Flash
  • Recipe C: Gemma 3 27B
  • Build: llama.cpp CUDA
  • Recipe D: Qwen3.6 27B
Important: -c and -np are coupled. In llama-server, -c is the total KV cache, shared evenly across all -np parallel slots. Each slot gets -c / -np tokens of context. Example: -c 131072 -np 4 = 32K per slot, not 128K. To give N users X context each: set -c X*N -np N. The recipes below specify per-slot context where ambiguous.

Recipe A: "Daily Driver" — Llama 3.1 70B (Q4_K_M) for 2 Users

Model max context: 128K. This recipe serves 4 users × 32K each (-c 131072 -np 4) across all 4 P40s — ~42 GB weights + ~42 GB KV cache = ~84 GB total. See the max-context option below for single-user 128K. 
# Start the server with a systemd service
# /etc/systemd/system/llama-server.service
# -c 131072 / -np 4 = 32K per slot (128K total KV cache)
[Unit]
Description=llama.cpp Inference Server
After=network.target

[Service]
Type=simple
User=llama
WorkingDirectory=/opt/llama.cpp
ExecStart=/opt/llama.cpp/build/bin/llama-server \
  -m /models/Llama-3.1-70B-Instruct-Q4_K_M.gguf \
  -ngl all \
  --split-mode layer \
  --tensor-split 22,22,24,24 \
  -c 131072 \
  -np 4 \
  -fa \
  --host 0.0.0.0 \
  --port 8080 \
  -b 512 \
  -ub 512 \
  --mlock
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target

# Enable and start:
sudo systemctl enable --now llama-server

Recipe B: "Next-Gen MoE" — DeepSeek-V4-Flash (IQ1_M-XL) for 1 User

Model max context: 1M tokens native. DeepSeek-V4-Flash (284B total / 13B activated MoE) replaces V3 (671B / 37B). Its Hybrid Attention (CSA + HCA) uses ~10% of V3.2's KV cache at equivalent context — the IQ1_M-XL GGUF (~63 GiB) fits all 4 P40s with ~28 GB free for KV cache. Single-user 128K context is easily achievable; 256K–384K is viable depending on batch size and thinking budget. DeepSeek recommends 384K context window minimum for Think Max mode. V4-Flash scores 86.2 MMLU-Pro, 88.1 GPQA Diamond, 91.6 LiveCodeBench in Think Max.
Requires a custom llama.cpp fork. V4 architecture is not in upstream llama.cpp yet. Use cchuter/llama.cpp @ feat/v4-port-cuda. This fork includes V4-specific CUDA kernels (CSA rope, sinkhorn, FP8 KV quantize, hyper-connection ops). Pascal (SM 6.1) uses software-emulated FP8 — no native FP8 hardware — so V4 ops run as emulated fallbacks. Functional but slower than Ada/Blackwell. See the build section below for Pascal-specific cmake flags.

Recipe B — Recommended: 128K Context, Single User (IQ1_M-XL)

# DeepSeek-V4-Flash IQ1_M-XL GGUF (~63 GiB, fully GPU-resident)
# 4 P40s (96 GB VRAM): weights ~63 GiB + KV cache ~4–8 GB at 128K = ~70 GiB total
# V4's CSA+HCA hybrid attention compresses KV dramatically — much more context-
# efficient than traditional dense or MoE attention.
# -c 131072 / -np 1 = 128K per slot. Single user with deep reasoning headroom.

llama-server \
  -m /models/DeepSeek-V4-Flash-IQ1_M-XL-00001-of-00002.gguf \
  --n-gpu-layers 999 \
  --split-mode layer \
  --tensor-split 24,24,24,24 \
  --ctx-size 131072 \
  --parallel 1 \
  --flash-attn on \
  --no-repack \
  --jinja \
  --reasoning think \
  --host 0.0.0.0 \
  --port 8080 \
  -b 512 \
  -ub 512 \
  --mlock \
  --temp 1.0 \
  --top-p 1.0

Build: V4-aware llama.cpp Fork for Pascal (SM 6.1)

# Clone the V4-aware fork — NOT upstream ggml-org/llama.cpp
git clone -b feat/v4-port-cuda https://github.com/cchuter/llama.cpp llama.cpp-v4
cd llama.cpp-v4

# Pascal (SM 6.1) uses software-emulated FP8 path. No native FP8 hardware.
# GGML_SCHED_MAX_SPLIT_INPUTS=128 required for V4 multi-GPU graph splits
# (V4's dense per-layer inputs exceed upstream default of 30 at split boundaries)
cmake -B build \
  -DGGML_CUDA=ON \
  -DCMAKE_CUDA_ARCHITECTURES="61" \
  -DCMAKE_CXX_FLAGS=-DGGML_SCHED_MAX_SPLIT_INPUTS=128 \
  -DGGML_OPENMP=ON

cmake --build build --config Release -j$(nproc)

# Verify CUDA backend and GPUs:
./build/bin/llama-cli --list-devices
# Should list all 4 Tesla P40 GPUs with ~24 GB each

# Download IQ1_M-XL quant (2 shards, ~63 GiB total):
# Option A: via huggingface-cli
pip install huggingface_hub[hf_transfer]
hf download teamblobfish/DeepSeek-V4-Flash-GGUF \
  --include "IQ1_M-XL/*" \
  --local-dir /models/

# Option B: direct download (replace with actual HF raw URLs)
# See https://huggingface.co/teamblobfish/DeepSeek-V4-Flash-GGUF for mirrors

# Install to system (optional):
sudo cmake --install build
Important flags for V4:
--no-repack — Required. Without it, the repack codepath doesn't release the source mmap, causing ~2× peak RAM at load (~120+ GB — would blow past your 64 GB system RAM).
--jinja — Enables DSML chat template baked into the GGUF. Tool calls return as proper tool_calls JSON.
--reasoning think — Enables Think High mode for complex problem-solving. Use --reasoning off for fast non-reasoning chat. Think Max requires a special system prompt (see DeepSeek docs).
--temp 1.0 --top-p 1.0 — DeepSeek's recommended sampling parameters for V4 series.
KV cache is always f16 — V4's K is already FP8-quantized at write time. --cache-type-k q8_0 is silently overridden to f16 by the fork.

Recipe C: "Small & Fast" — Gemma 3 27B (Q5_K_M) for Interactive Chat

Model max context: 128K. This recipe runs at 32K on 2 P40s with speculative decoding. Full 128K needs ~42 GB (18 GB weights + 24 GB KV). 2 P40s (48 GB) works but is tight; 3–4 P40s comfortably keep the draft model for 2–3× speedup. Single P40 max: ~32K. 
# A smaller model that fits entirely in 2 P40s (48 GB)
# Use remaining 2 GPUs for speculative decoding draft model
# -c 32768 / -np 8 = 4K per slot (quick interactive chat, low context per user)

llama-server \
  -m /models/gemma-3-27b-it-Q5_K_M.gguf \
  -md /models/gemma-3-1b-it-Q8_0.gguf \
  -ngl all \
  --split-mode layer \
  --tensor-split 24,24,0,0 \
  -c 32768 \
  -np 8 \
  -fa \
  --host 0.0.0.0 \
  --port 8080 \
  -b 1024 \
  -ub 1024

Speculative decoding trick: The -md flag loads a small draft model (Gemma 1B) that predicts tokens, the main model verifies. This can 2–3× token generation speed for negligible VRAM cost (~2 GB for the draft model).

Building llama.cpp with CUDA for P40

# Build llama.cpp with CUDA support for Pascal GPUs
git clone https://github.com/ggml-org/llama.cpp
cd llama.cpp
mkdir build && cd build

# P40s are SM 6.1 — use CUDA_ARCHITECTURES=61
cmake .. \
  -DGGML_CUDA=ON \
  -DCMAKE_CUDA_ARCHITECTURES="61" \
  -DGGML_CUDA_FORCE_MMQ=ON \
  -DGGML_CUDA_NCCL=OFF \
  -DGGML_CUDA_PEER_MAX_BATCH_SIZE=128 \
  -DGGML_OPENMP=ON

cmake --build . --config Release -j$(nproc)

# Verify CUDA backend is active:
./bin/llama-cli --list-devices
# Should list all 4 Tesla P40 GPUs with ~24 GB each

# Install to system (optional):
sudo cmake --install .

Recipe D: Qwen3.6 27B — Dense Production Serving for 4+ Users

Model max context: 262K native (up to 1M with YaRN). Option 4 below serves the full 262K across 4 P40s. Options 1–3 use lower contexts for higher concurrency. Gated DeltaNet layers (48 of 64) use O(1) state — KV cache only grows for the 16 standard-attention layers, making long context unusually cheap on VRAM.

Qwen3.6-27B is a 27B dense model (not MoE) with a hybrid Gated DeltaNet + Gated Attention architecture. At BF16 it's ~54 GB — fits on 3 P40s (tensor-parallel) or all 4 with room for long-context KV cache. Unlike Mixtral's expert routing overhead, this is a pure dense model: simpler to serve, more deterministic latency, and handles high concurrency well with vLLM's continuous batching.

Why Qwen3.6-27B shines on P40s:
At Q4_K_M GGUF (~17 GB) it fits on a single P40. At BF16 it needs 3 P40s. The Gated DeltaNet linear attention heads are compute-efficient — fewer quadratic-attention bottlenecks on Pascal's limited FP16 throughput. Multimodal: images and video via vision encoder. Strong coding/agent scores (SWE-bench Verified 77.2%, AIME 2026 94.1%). Text-only mode (--language-model-only) drops the vision encoder to save ~2 GB VRAM. 
# Option 1: GGUF Q4_K_M via llama.cpp — fits on 1–2 P40s
llama-server \
  -m /models/Qwen3.6-27B-Q4_K_M.gguf \
  -ngl all \
  --split-mode layer \
  --tensor-split 24,24,0,0 \
  -c 65536 \
  -np 8 \
  -fa \
  --host 0.0.0.0 \
  --port 8080 \
  -b 512 \
  -ub 512

# Option 2: vllm-pascal with AWQ quantized (~13 GB, fits 2 P40s, room for 128K context)
docker run -d --name vllm-qwen36 \
  --gpus all \
  -p 8000:8000 \
  -v /models:/models \
  ghcr.io/sasha0552/vllm:v0.10.0 \
  --model /models/Qwen3.6-27B-AWQ \
  --tensor-parallel-size 2 \
  --max-model-len 131072 \
  --gpu-memory-utilization 0.85 \
  --max-num-seqs 8 \
  --reasoning-parser qwen3 \
  --language-model-only

# Option 3: vllm-pascal with MTP speculative decoding for 1.3–1.5× speedup
docker run -d --name vllm-qwen36 \
  --gpus all \
  -p 8000:8000 \
  -v /models:/models \
  ghcr.io/sasha0552/vllm:v0.10.0 \
  --model /models/Qwen3.6-27B-AWQ \
  --tensor-parallel-size 2 \
  --max-model-len 65536 \
  --gpu-memory-utilization 0.85 \
  --max-num-seqs 8 \
  --reasoning-parser qwen3 \
  --speculative-config '{"method":"qwen3_next_mtp","num_speculative_tokens":2}'

# Option 4: Full native 262K context — AWQ across 4 P40s (recommended for long-context tasks)
# Model ~13 GB + KV cache ~17 GB = ~30 GB base. 4 P40s (96 GB) leave ~66 GB for
# KV cache, supporting multiple concurrent users at 262K context or using prefix caching.
# Qwen3.6's Gated DeltaNet layers (48 of 64) use linear attention — their state
# is O(1) per layer, not O(n). Only 16 standard-attention layers grow KV cache
# with sequence length, making 262K context feasible on 96 GB total VRAM.
docker run -d --name vllm-qwen36-fullctx \
  --gpus all \
  -p 8000:8000 \
  -v /models:/models \
  ghcr.io/sasha0552/vllm:v0.10.0 \
  --model /models/Qwen3.6-27B-AWQ \
  --tensor-parallel-size 4 \
  --max-model-len 262144 \
  --gpu-memory-utilization 0.90 \
  --max-num-seqs 4 \
  --reasoning-parser qwen3 \
  --enable-prefix-caching \
  --language-model-only
Pascal caveat: The Qwen3.6 Gated DeltaNet uses linear attention heads (48 heads for V, 16 for QK at dim 128). These operate as dense matmuls — no flash-attention speedup on Pascal without tensor cores. Performance will be memory-bandwidth bound (~347 GB/s per P40). For maximum throughput, use AWQ or GGUF quantization to reduce memory pressure. AWQ at ~13 GB on 2 P40s leaves ample VRAM for long KV caches and high concurrency.
