Critical Limitations

  • NO TENSOR CORES — Pascal predates Volta. FP16 is not accelerated.
  • CC 6.1 — Official vLLM & many modern frameworks require CC ≥ 7.0. But pascal-pkgs-ci backports vLLM + Triton to Pascal (see Section 5).
  • PASSIVE — No fan; requires custom cooling in a non-server chassis.
  • GDDR5 — Slower than HBM2 in V100/P100.
  • 96 GB VRAM — Combined 4× is excellent for quantized models.
  • INT8 — 47 TOPS is very usable for quantized inference.
Framework compatibility: While official vLLM requires CC ≥ 7.0, the community project pascal-pkgs-ci backports vLLM (up to v0.10.0+), Triton (up to v3.4.0), and aphrodite-engine to Pascal SM 6.1. This gives you access to PagedAttention, continuous batching, prefix caching, FP8 KV cache, tensor/pipeline/expert parallelism, and the full OpenAI-compatible API — all on P40s. With this, your software options expand dramatically beyond llama.cpp alone.

2. Platform & Motherboard Selection

The Core Constraint: PCIe Lanes

Each P40 wants PCIe 3.0 ×16 (electrically). For four cards you need 64 PCIe lanes minimum plus lanes for NVMe storage and chipset I/O. This rules out all consumer desktop platforms (LGA 1700, AM5, etc.) which top out at 20–28 CPU lanes.

Power Supply

4× P40 at 250W = 1,000W for GPUs alone. Add 200W for CPU + motherboard + fans. Minimum 1,500W PSU recommended. Look for:

  • Corsair HX1500i — 1500W, Platinum, fully modular, available at Scorptec/PCCG (~AU$529).
  • be quiet! Dark Power Pro 13 1600W — Titanium efficiency, available at Scorptec (~AU$599).

EPS 12V adapter needed: P40s use an 8-pin EPS connector (CPU-power type), not PCIe 8-pin. Buy dual PCIe 6+2 to EPS 8-pin adapter cables. Do NOT force a PCIe 8-pin into the EPS socket — the pinout is different.

3. Cooling the Passive P40s

The Tesla P40 has no active fan. In a data center, chassis fans handle cooling. In a home build you must add forced airflow.

Option A: 3D-Printed Shroud + High-RPM Fan (Recommended)

# STL files available on Printables/Thingiverse — search "Tesla P40 fan shroud"
# Fan: 2× 40mm server fans (Delta EFB0412VHD or similar, 15,000+ RPM)
# Connect via motherboard fan header with PWM control
# Total cost per GPU: ~AU$25 for shroud + $15 for fans

Option B: Mining Rig Open-Air Frame

Mount GPUs on an open-air mining frame with a high-CFM box fan or a rack of 120mm fans blowing across all four cards. Effective but loud.

Option C: 4U Server Chassis

A Rosewill RSV-L4500U or Supermicro 846 chassis with internal 80mm fan wall. These chassis are designed for passive GPUs with high-static-pressure mid-wall fans. Approx. AU$250–400 used.

Never run a P40 without active cooling. It will thermal-throttle at ~85°C and may shut down. Hardware damage is possible.

5. pascal-pkgs-ci — vLLM & Triton Backported to Pascal

sasha0552/pascal-pkgs-ci (195 ★, 32 forks) is a community CI pipeline that rebuilds vLLM and Triton with Pascal SM 6.1 compatibility. It patches the CUDA architecture checks to accept sm_61 (and sm_60 for P100), providing prebuilt wheels and Docker images.

What's Available

PackageVersionsInstall As
vLLM 0.5.5 → 0.10.0 + nightly (daily builds from main) vllm-pascal
Triton 2.2.0 → 3.4.0 triton-pascal
Aphrodite Engine Supported via extra-index aphrodite-engine
Caveat (v0.7.0+): Pascal builds are Pascal-only — they strip non-Pascal GPU architectures to reduce binary size. For a heterogeneous cluster, use the official vLLM for non-Pascal GPUs and vllm-pascal for Pascal GPUs, connecting them via tensor or pipeline parallelism.

Manual pip Installation

# 1. Add the Pascal package index
export PIP_EXTRA_INDEX_URL="https://sasha0552.github.io/pascal-pkgs-ci/"

# 2. Create and activate a virtual environment
python -m venv vllm-pascal-env
source vllm-pascal-env/bin/activate

# 3. Install vLLM for Pascal
pip install vllm-pascal==0.10.0   # omit version for nightly

# 4. Install transient-package (handles triton→triton-pascal swap)
pipx install transient-package

# 5. Swap in the Pascal-compatible Triton
transient-package install \
  --interpreter vllm-pascal-env/bin/python \
  --source triton \
  --target triton-pascal

# 6. Patch PyTorch's scheduler to accept SM 6.x
sed -e "s/.major < 7/.major < 6/g" \
    -e "s/.major >= 7/.major >= 6/g" \
    -i vllm-pascal-env/lib/python3.*/site-packages/torch/_inductor/scheduler.py \
       vllm-pascal-env/lib/python3.*/site-packages/torch/utils/_triton.py

# 7. Launch!
vllm serve /models/Mixtral-8x22B-Instruct-v0.1 \
  --tensor-parallel-size 4 \
  --dtype float16 \
  --max-model-len 16384

What This Unlocks for P40s

vLLM Features on P40

  • PagedAttention — efficient KV cache management
  • Continuous Batching — interleave requests from multiple users
  • Prefix Caching — reuse KV cache for repeated prompts
  • Tensor Parallelism — split layers across 4 GPUs
  • Pipeline Parallelism — distribute across GPUs with slow interconnects
  • Expert Parallelism — MoE expert distribution across GPUs
  • OpenAI-compatible API — drop-in replacement
  • FP8 KV Cache — depends on model support

GPUs with Pascal

  • Tesla P40 — 24 GB (the focus of this guide)
  • Tesla P100 — 16 GB HBM2 (faster, less VRAM)
  • Quadro GP100 — 16 GB HBM2
  • Titan Xp — 12 GB
  • GTX 1080 Ti — 11 GB
Pascal Triton performance note: Triton kernels on Pascal (SM 6.1) lack the hardware features of Volta+ — specifically, no tensor cores, no FP16 matrix multiply acceleration, and limited shared memory. Triton kernels work but won't match Volta+ speed. For pure INT8 quantized inference, llama.cpp may still be faster. For FP16 models, vllm-pascal + PagedAttention is the best option.

Two great options, different strengths:

vllm-pascal (via pascal-pkgs-ci) — Best for production multi-user serving. PagedAttention + continuous batching gives the highest throughput. Use for FP16/BF16 models, MoE models with expert parallelism, and when you need the OpenAI-compatible API with streaming.

llama.cpp — Best for quantized GGUF models. Broader model support, speculative decoding, CPU+GPU hybrid for oversized models. Use for Q4_K_M/Q5_K_M GGUF models, when VRAM is tight, or when you need CPU fallback for giant MoE models.

8. MoE Partial Expert Offloading

Mixture-of-Experts models have many total parameters but only a subset is active per token. This is the key to fitting massive models on limited VRAM. The full model can live across GPU VRAM + CPU RAM (or SSD), and only the active experts are loaded.

Strategy A: llama.cpp with --no-kv-offload + Layer Splitting

llama.cpp does not have native "expert-aware" offloading yet. However, you can:

  1. Put shared layers (attention, embeddings) on GPU via --n-gpu-layers
  2. Let expert FFN layers spill to CPU RAM when VRAM runs out
  3. The MoE router only activates ~5–10% of experts, so CPU overhead is manageable
# Example: DeepSeek-V3 GGUF quantized (IQ3_XXS or Q3_K_M)
# Total GGUF size: ~250 GB at Q3_K_M for 671B params
# Shared layers (~30B params) + top experts cached on GPU
# Remaining experts paged from system RAM (128 GB DDR4 recommended)

llama-server \
  -m DeepSeek-V3-Q3_K_M.gguf \
  -ngl 999 \
  --split-mode layer \
  --tensor-split 24,24,24,24 \
  -c 8192 \
  -fa \
  --no-kv-offload
How it works: -ngl 999 tries to push all layers to GPU. When VRAM is full, llama.cpp automatically falls back to CPU for remaining layers. Since MoE experts are in later layers, they spill to CPU. The GPU handles attention and shared FFN; CPU handles expert routing. With 128 GB system RAM you can load the entire 671B model.

Strategy B: KTransformers (Specialized MoE Offloading)

KTransformers is purpose-built for running giant MoE models with limited GPU VRAM. It uses CUDA graphs with CPU offload and supports DeepSeek-V2/V3 natively. 

# Install KTransformers
git clone https://github.com/kvcache-ai/ktransformers
cd ktransformers
pip install -e .

# Run DeepSeek-V3 with expert offloading
# Shared layers on GPU, expert FFNs on CPU
python -m ktransformers.local_chat \
  --model_path /path/to/DeepSeek-V3 \
  --gguf_path /path/to/DeepSeek-V3-Q4_K_M.gguf \
  --optimize_rule_path ktransformers/optimize/optimize_rules/DeepSeek-V3-Chat.yaml

Strategy C: llama-swap for Dynamic Model Swapping

llama-swap is a transparent proxy that automatically loads/unloads models in llama-server. You can run a smaller dense model (e.g. Llama 3.1 8B) for quick responses and swap to a large MoE model on demand. 

# config.yaml for llama-swap
models:
  small:
    cmd: llama-server -m llama-3.1-8b-Q4_K_M.gguf -ngl 99 -c 4096 --port 8081
    proxy: http://localhost:8081
  big:
    cmd: llama-server -m DeepSeek-V3-Q3_K_M.gguf -ngl 999 -c 8192 --port 8082
    proxy: http://localhost:8082
    unload:
      after_idle: 300  # Unload after 5 min idle

9. Multi-User Concurrent Serving

llama.cpp's llama-server supports concurrent requests via parallel slots and continuous batching. Here's how to configure for 2+ simultaneous users:

8.1 llama-server Multi-User Configuration

# Start server with 4 parallel slots and generous context
llama-server \
  -m Mixtral-8x22B-Q4_K_M.gguf \
  -ngl 999 \
  --split-mode layer \
  --tensor-split 24,24,24,24 \
  -c 32768 \
  -np 4 \
  -fa \
  --host 0.0.0.0 \
  --port 8080 \
  -b 512 \
  -ub 512 \
  --mlock \
  --no-mmap

Key Flags Explained:

FlagMeaning
-c 32768Total KV cache context size. Split across -np slots → ~8K per user.
-np 4Parallel request slots. Serve up to 4 concurrent users/requests.
-b 512Physical batch size for prompt processing (prefill).
-ub 512Physical batch size for token generation (decode).
--mlockLock model in RAM to prevent swapping.
--no-mmapLoad entire model into RAM first (prevents disk I/O stalls).

8.2 Performance Under Concurrent Load

With 2 users, each gets their own KV cache slot and the server interleaves token generation. With 4 P40s on pipeline-parallel (--split-mode layer), tokens from different users flow through the pipeline simultaneously — this is where pipeline parallelism shines. 

# Test concurrent load with 2 simultaneous curl requests:
curl -X POST http://server:8080/v1/chat/completions \

  &

8.3 Reverse Proxy / Load Balancer (Optional)

For a more polished home-network setup, put nginx or Caddy in front of llama-server with rate limiting and access control: 

# /etc/nginx/sites-available/llm-proxy
server {
    listen 443 ssl;
    server_name llm.home.arpa;

    ssl_certificate     /etc/ssl/llm.crt;
    ssl_certificate_key /etc/ssl/llm.key;

    location /v1/ {
        proxy_pass http://127.0.0.1:8080;
        proxy_http_version 1.1;
        proxy_set_header Connection "";
        proxy_buffering off;
        proxy_read_timeout 300s;

        # Rate limit: 20 requests per minute per IP
        limit_req zone=llm burst=5 nodelay;
    }
}

10. Streaming Models Larger Than Total VRAM

Even with 96 GB VRAM, some models (DeepSeek-V3, Llama 3.1 405B, GPT-OSS) far exceed available memory. Here are the strategies, from simplest to most advanced:

Strategy 1: CPU+GPU Hybrid (llama.cpp built-in)

This is the simplest approach. llama.cpp automatically spills layers to CPU RAM when GPU VRAM is exhausted. With 128 GB system RAM, you can run massive models. 

# Run a 400B+ model: load what fits on GPUs, rest on CPU
llama-server \
  -m Llama-3.1-405B-Q3_K_M.gguf \
  -ngl 999 \
  --split-mode layer \
  --tensor-split 24,24,24,24 \
  -c 4096 \
  -np 2 \
  -fa
Expect ~1–3 tokens/sec when most layers run on CPU. This is usable for background batch processing but frustrating for interactive chat. For interactive use, prefer MoE models with partial expert loading (Section 7).

Strategy 2: RPC Backend (Multi-Machine Split)

llama.cpp supports an RPC backend that lets you split a model across multiple machines on the same network. You could add a second machine with more GPUs (or just CPU RAM) to offload additional layers. 

# On the "server" machine (with GPUs):
llama-server -m model.gguf --rpc $SERVER_IP:50052

# On the "worker" machine (with extra RAM/GPUs):
cd llama.cpp && ./build/bin/llama-rpc-server -H $SERVER_IP -p 50052

Strategy 3: Row-Level Tensor Splitting (Deprecated)

The older --split-mode row in llama.cpp splits individual tensor rows across GPUs but has been superseded by --split-mode tensor (experimental). For P40s without NVLink, layer mode is recommended.

Strategy 4: Expert-Aware Streaming (KTransformers)

KTransformers can dynamically page individual MoE expert weights from SSD/NVMe into GPU VRAM as the router activates them. This is the most advanced approach and yields the best interactive performance for giant MoE models on limited VRAM. 

# KTransformers with NVMe expert offloading
# Requires a fast NVMe SSD (PCIe 4.0 recommended, P40s support 3.0)
# Expert weights are mmap'd from NVMe and cached in GPU as needed

python -m ktransformers.local_chat \
  --model_path /mnt/nvme/DeepSeek-V3 \
  --optimize_rule_path ktransformers/optimize/optimize_rules/DeepSeek-V3-Chat.yaml \
  --cpu_infer 32 \
  --cache_lens 8192 \
  --max_new_tokens 2048

Thermal Management Script

#!/bin/bash
# /usr/local/bin/gpu-fan-check.sh
# Monitor GPU temps and alert if too high
THRESHOLD=75
while true; do
  TEMPS=$(nvidia-smi --query-gpu=temperature.gpu --format=csv,noheader)
  for temp in $TEMPS; do
    if [ "$temp" -gt "$THRESHOLD" ]; then
      echo "[WARN] GPU temp $temp°C exceeds $THRESHOLD°C at $(date)" >> /var/log/gpu-temp.log
    fi
  done
  sleep 10
done

#!/bin/bash
# /usr/local/bin/switch-model.sh

    NP="4"
    ;;
  mixtral)

    NP="2"
    ;;
  deepseek)
    MODEL_PATH="/models/DeepSeek-V3-IQ3_XXS.gguf"
    CTX="8192"
    NP="2"
    ;;
esac

[Service]
ExecStart=
ExecStart=/opt/llama.cpp/build/bin/llama-server \

  -ngl all \

  --tensor-split 22,22,24,24 \
  -c $CTX \
  -np $NP \
  -fa \
  --host 0.0.0.0 \
  --port 8080 \
  -b 512 -ub 512 \
  --mlock
EOF

sudo systemctl start llama-server
echo "Switched to $MODEL model"
