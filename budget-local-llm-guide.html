Why Run Large Models Locally?

Running 70B+ parameter models at home means no API costs, no rate limits, complete privacy, and the ability to customize. The barrier has dropped dramatically thanks to quantization (GGUF, AWQ, GPTQ), efficient inference engines (llama.cpp, vLLM, ExLlamaV2), and a thriving used-GPU market.

DeepSeek-V4-Flash is already released (it's powering this conversation). As an MoE model with Multi-head Latent Attention, it represents the cutting edge of what's possible — and with the right hardware and quantization, running models at this scale at home is increasingly achievable.

The golden rule: VRAM is everything. A 70B model in 4-bit needs ~40-48 GB. A 120B model needs ~68-80 GB. MoE models like Mixtral 8x7B or DeepSeek-V4 need less compute but similar memory since all experts are loaded.

VRAM Requirements Reference

Formula: VRAM ≈ params × bytes_per_param × 1.2 (20% overhead for KV cache, activations).

AWQ (Activation-Aware Weight Quantization)

Used by vLLM, TGI, AutoAWQ. 4-bit only typically. Slightly better quality than naive round-to-nearest 4-bit. Requires calibration dataset. Good for serving.

GPTQ (Post-Training Quantization)

Older standard, popularized by AutoGPTQ. 4-bit. Works well with ExLlamaV2. Requires calibration. Slightly slower than AWQ for batch inference.

GGUF (GGML Universal Format)

The standard for llama.cpp. Supports all quantization levels (1.5 to 8 bit). Runs on CPU, GPU, or hybrid. Most flexible format for home users. Massive ecosystem of pre-quantized models on Hugging Face.

Bitsandbytes (NF4 / FP4)

Hugging Face native 4-bit. Uses double quantization for extra savings. Good for transformers integration. Slower than GGUF for inference.

GGUF Advantages

  • CPU+GPU hybrid inference (offload partial layers)
  • Widest quantization selection (Q2–Q8, IQ1–IQ4)
  • No Python/transformers dependency needed
  • Best memory efficiency per quality
  • Huge HF hub ecosystem of quantized models

GGUF Limitations

  • Single-user, not designed for serving
  • No dynamic batching (but added in llama-server)
  • Slower than vLLM for high-throughput serving
  • Not ideal for multi-user API serving
Recommendation: Start with Q4_K_M GGUF for home use. If you have VRAM to spare, step up to Q5_K_M or Q6_K. If VRAM-constrained, Q3_K_M or Q2_K still deliver remarkable quality for a 70B model.

Software Stack

Inference Engines

llama.cpp

Home #1

  • C/C++, no dependencies
  • GGUF format (all quants)
  • CPU + GPU hybrid (CUDA/Metal/Vulkan)
  • llama-server: OpenAI-compatible API
  • Best memory efficiency
  • SWA, speculative decoding, multimodal

vLLM

Serving

  • PagedAttention for KV cache efficiency
  • Continuous batching (high throughput)
  • OpenAI-compatible API
  • Supports AWQ, GPTQ, FP8
  • Needs CUDA GPU with decent VRAM
  • Best for multi-user or chat apps

ExLlamaV2

Fast single-user

  • Optimized CUDA kernels for GPTQ/AWQ
  • Fastest single-user token generation
  • Supports 4-bit and 8-bit
  • Good for chat/roleplay use cases
  • Less flexible than llama.cpp

Ollama

Easiest setup

  • Wraps llama.cpp in pleasant UX
  • ollama pull model — one command
  • Model library, modelfiles
  • OpenAI-compatible API
  • Good for beginners

Text Generation WebUI

Feature-rich

  • Gradio web UI (a1111-style)
  • Backends: llama.cpp, ExLlama, AutoGPTQ, Transformers
  • LoRA loading, multimodal, extensions
  • Good for experimentation

Setting Up llama.cpp (Quickstart)

# Build from source (Linux/macOS)
git clone https://github.com/ggml-org/llama.cpp
cd llama.cpp
make -j

# Or CMake with CUDA support
cmake -B build -DGGML_CUDA=ON
cmake --build build --config Release -j

# Download a 70B GGUF model (example)
wget https://huggingface.co/bartowski/Llama-3.1-70B-Instruct-GGUF/resolve/main/Llama-3.1-70B-Instruct-Q4_K_M.gguf

# Run with GPU offload (24 layers to GPU)
./llama-cli -m Llama-3.1-70B-Instruct-Q4_K_M.gguf \
  -ngl 24 -n 512 -p "Hello, explain quantum computing"

# Start API server
./llama-server -m Llama-3.1-70B-Instruct-Q4_K_M.gguf \
  -ngl 48 --port 8080 --host 0.0.0.0

Setting Up vLLM for Serving

# Install
pip install vllm

# Serve a model
python -m vllm.entrypoints.openai.api_server \
  --model meta-llama/Llama-3.1-70B-Instruct \
  --quantization awq \
  --dtype half \
  --max-model-len 8192 \
  --tensor-parallel-size 2  # if 2 GPUs

# Or with a local AWQ quant
python -m vllm.entrypoints.openai.api_server \
  --model /path/to/model-awq \
  --max-model-len 4096

DeepSeek-V4-Flash — Now Available

DeepSeek-V4-Flash is released and running right now — it's the model powering this conversation (model ID: deepseek/deepseek-v4-flash). It is an MoE architecture from DeepSeek, continuing the lineage of V2 → V3 → R1. Like its predecessors, it uses Multi-head Latent Attention (MLA) for memory-efficient KV cache, and a sparse MoE structure where only a fraction of parameters activate per token. The "Flash" suffix indicates optimizations for inference speed and reduced memory footprint.

While exact architecture specs (total parameter count, expert count, active params) are not publicly documented at this time, the architecture follows DeepSeek's established MoE+MLA blueprint. Here's what we can infer from the lineage:

  • Architecture: Mixture-of-Experts with Multi-head Latent Attention
  • MLA advantage: Compresses KV cache by ~75% vs standard MHA — dramatically reduces memory for long contexts
  • Sparse activation: Only a subset of experts activate per token, keeping compute costs far below what total parameters would suggest
  • Inference optimization: The "Flash" variant suggests further kernel-level and/or quantization optimizations for faster generation
  • GGUF community ports: Check Hugging Face for community-quantized GGUF versions as they appear

For DeepSeek V3/R1 (671B) — the previous generation and likely similar in size to V4-Flash — here's what's needed at home:

  • GGUF Q2_K: ~249 GB — needs 10x P40 or 5x 3090
  • GGUF Q3_K_M: ~315 GB — needs 13x P40 or 7x 3090
  • GGUF Q4_K_M: ~403 GB — needs 17x P40 or 8x 3090
  • IQ2 via imatrix: ~160 GB — fits on 7x P40 or 3-4x 3090

MoE-Specific Optimization Tips

  • llama.cpp: Use -ngl N to offload layers to GPU. For MoE, you can offload fewer layers since compute per token is lower.
  • vLLM: MoE models benefit from --tensor-parallel-size to distribute experts across GPUs.
  • KV cache: DeepSeek's MLA means dramatically smaller KV cache. Set -c 16384 or higher — it won't cost as much VRAM as dense models.
  • Expert offloading: llama.cpp can offload inactive experts to RAM and bring active ones to GPU — experimental but promising for MoE.

What's "Usable"?

  • > 30 tok/s: Feels instantaneous — like ChatGPT
  • 15–30 tok/s: Comfortable real-time chat
  • 8–15 tok/s: Acceptable for chat, fine for batch
  • 3–8 tok/s: Noticeable delay, OK for background tasks
  • < 3 tok/s: Painful for chat, usable for batch/offline
Prompt processing (ingesting your input) is usually 2-5x faster than token generation. Long prompts (>4K tokens) can take several seconds even on fast hardware. MoE models have faster prompt processing since fewer expert parameters are computed per token.

FAQ & Pro Tips

Should I buy P40s or save for 3090s?

P40s are the best value per GB of VRAM (~$10/GB). 3090s offer 3-4x more throughput per GB. If you're budget-constrained and patient (3-8 tok/s is OK), go P40. If you want comfortable chat speeds, save for 3090s.

Can I mix different GPUs?

Yes, llama.cpp supports heterogeneous GPUs. You can have a 3090 + P40, for example. The faster GPU will handle more layers. Use --tensor-split to control distribution. The main constraint: all GPUs must be NVIDIA (or all AMD for ROCm).

Do I need NVLink between 3090s?

No. NVLink helps marginally (~5-10%) with multi-GPU inference but is not required. PCIe 3.0/4.0 bandwidth is sufficient for LLM inference since layer outputs are relatively small.

How much system RAM do I need?

At least 32 GB. For CPU offloading or running CPU-only, 64-128 GB recommended. For models split across GPUs, system RAM is less critical (16-32 GB is fine).

Should I use llama.cpp or vLLM at home?

For single-user: llama.cpp (or Ollama on top). For multi-user or API serving: vLLM. For fastest single-user speed: ExLlamaV2.

Where do I find pre-quantized models?

Hugging Face: search for GGUF combined with your model name. Popular quantizers: TheBloke, bartowski, mradermacher, hugging-quants. Most now provide an imatrix (importance matrix) for higher-quality quantizations.

What about AMD GPUs?

ROCm support in vLLM and SGLang is good. ROCm + llama.cpp has improved but is still behind CUDA. The RX 7900 XTX (24 GB) is a solid option if you want to avoid NVIDIA pricing. Expect ~80% of RTX 4090 performance.

Can I run 405B (Llama 3.1) at home?

Yes, but only with heavy quantization:

  • Q2_K: ~150 GB → 6x P40 (~$600) or 7x 3090 if you want speed
  • IQ2: ~130 GB → 5-6x P40
  • IQ1_S: ~90 GB (experimental, quality loss noticeable)
A Mac Studio M2 Ultra with 192 GB is the most practical way to run 405B at home without multi-GPU headaches.

Power consumption?

  • 1x RTX 3090 at load: ~350W
  • 2x RTX 3090: ~700W (need 1000W+ PSU)
  • 1x P40 at load: ~250W
  • 4x P40: ~1000W
  • Mac M2 Ultra: ~150W (much more efficient)
